# Skill 1_10：从原子任务到精炼原型设计文档

## 概述

- **所属阶段**：S2「原型评审 / 落地规划」
- **目标**：基于原子任务文档（来自 `pdd-skill1_7-proto2planning` 和 `pdd-skill1_8-proto2planning`），生成精炼的原型设计文档，适合开会汇报、向利益相关者展示或作为项目概览。
- **执行者**：Cursor（Technical Writer & Architect）
- **输入**：
  - 原子任务文档路径（如 `docs/plans/YYYY-MM-DD-atomic-tasks-for-prototype-design.md`）
  - 使用场景（可选，默认为"开会时的汇报"）
- **输出**：
  - 精炼的原型设计文档，保存到 `docs/plans/YYYY-MM-DD-<feature-name>-prototype-design.md`

---

## 定位与与其它 Skills 的关系

- **前置 Skill**：
  - `pdd-skill1_7-proto2planning`：分析代码更改，划分原子任务
  - `pdd-skill1_8-proto2planning`：补充原子任务内容
- **并行 Skill**：`brainstorming`、`writing-plans`
  - 本 Skill 采用 `brainstorming` 的对话式方法，逐步细化设计
  - 输出格式参考 `writing-plans` 的结构，但更精炼，适合汇报
- **后续 Skill**：
  - `pdd-skill1_9-proto2planning`：基于原子任务生成详细的实现规划（包含技术细节）

> **重要区别**：
>
> - `pdd-skill1_9`：详细的实现规划，包含技术细节、文件路径、验收标准（适合开发团队）
> - `pdd-skill1_10`：精炼的设计文档，突出价值、架构、功能（适合汇报、评审）

## Skill 内部工作流

> **角色**：Technical Writer & Architect  
> **目标**：生成精炼、清晰、适合汇报的原型设计文档，突出核心价值而非技术细节。

### 流程概述

采用 **对话式流程**，参考 `brainstorming` skill 的方法：

1. **理解阶段**：阅读原子任务文档，理解核心业务逻辑
2. **探索阶段**：根据使用场景，确定文档重点和详细程度
3. **呈现阶段**：分章节生成内容（每章 200-300 字），每章后确认
4. **文档化阶段**：将确认的内容写入文档

### 步骤 1：理解原子任务文档（Understanding Atomic Tasks）

1. **读取原子任务文档**：
   - 识别核心业务概念
   - 理解数据流向
   - 提取关键功能模块
   - 识别任务分组（A、B、C...）

2. **提取关键信息**：
   - **业务背景**：解决什么问题？为什么需要？
   - **核心价值**：带来什么好处？
   - **系统架构**：核心组件和数据流向
   - **关键功能**：主要功能模块（3-7 个）
   - **数据结构**：核心数据表（简化版）
   - **业务规则**：关键业务规则（3-5 条）

> **对话式提问**（如果需要澄清）：
>
> - "这个系统的核心价值是什么？解决了什么业务痛点？"
> - "有哪些关键功能模块？优先级如何？"

### 步骤 2：确定使用场景和文档重点（Determine Focus）

**根据使用场景调整文档重点**：

| 使用场景       | 重点内容                               | 详细程度 |
| -------------- | -------------------------------------- | -------- |
| 开会时的汇报   | 业务价值、架构、核心功能、实施计划     | 中等     |
| 向产品经理汇报 | 业务价值、用户价值、功能列表、时间计划 | 较高     |
| 向技术团队汇报 | 架构设计、技术方案、实施计划           | 较高     |
| 向管理层汇报   | 业务价值、ROI、风险、时间计划          | 较低     |
| 项目概览文档   | 全面但精炼，所有章节                   | 中等     |

**对话式确认**：
> "文档主要用于什么场景？是技术评审、产品评审，还是管理层汇报？"

**基于确认，确定文档结构**：

- **必须包含**：项目概述、系统架构、核心功能、实施计划
- **可选包含**：数据结构设计、关键业务规则、用户界面设计、验收标准、后续规划

### 步骤 3：生成文档内容（Generate Content）

**按章节生成，每章 200-300 字，每章后确认**：

#### 章节 1：项目概述

**四段式结构**（必须遵循）：

1. **定义**（What）：系统/功能是什么？
   - 一句话定义核心概念
   - 关键术语解释（如有必要）
   - 示例：版本映射表是什么？租户版本管理是什么？

2. **问题**（Why）：解决什么问题？
   - 业务痛点（2-3 条）
   - 现有系统的局限性
   - 典型场景举例

3. **解决**（How）：如何解决？
   - 核心解决方案（1-2 段）
   - 关键技术手段
   - 设计思路

4. **价值**（Value）：带来什么价值？
   - 核心价值点（3-5 条，用 ✅ 标记）
   - 业务收益
   - 技术收益

**内容要点**：

- 定义：50-100 字，清晰说明系统/功能是什么
- 问题：100-150 字，列出 2-3 个核心痛点
- 解决：100-150 字，说明解决方案
- 价值：100-150 字，列出 3-5 条核心价值

**呈现方式**：先呈现「项目概述」章节（包含四段式结构），询问："这个概述是否准确？四段式结构是否清晰？需要补充或调整吗？"

#### 章节 2：系统架构

**内容要点**：

- 核心组件（架构图或文字描述）
- 数据流向（简化流程图）
- 关键设计决策（2-3 条）

**呈现方式**：先呈现「系统架构」章节，询问："架构描述是否清晰？"

#### 章节 3：核心功能

**内容要点**：

- 列出 3-7 个核心功能模块
- 每个功能：功能描述（1 段）、关键规则（2-3 条）、典型场景（1 个）

**呈现方式**：先呈现「核心功能」章节，询问："功能列表是否完整？优先级是否正确？"

#### 章节 4：数据结构设计（可选）

**内容要点**：

- 核心数据表（简化版，只列出关键字段）
- 表之间的关系

**呈现方式**：如果使用场景需要技术细节，呈现此章节。

#### 章节 5：关键业务规则（可选）

**内容要点**：

- 列出 3-5 条关键业务规则
- 每条规则：规则名称、规则描述、典型场景

**呈现方式**：如果使用场景需要业务细节，呈现此章节。

#### 章节 6：用户界面设计（可选）

**内容要点**：

- 列出 2-3 个主要界面
- 每个界面：位置、功能、关键交互

**呈现方式**：如果使用场景需要 UI 细节，呈现此章节。

#### 章节 7：实施计划

**内容要点**：

- 阶段划分（3-5 个阶段）
- 每个阶段：阶段名称、时间估算、关键产出物
- 技术栈（简要）
- 风险与应对（表格）

**呈现方式**：先呈现「实施计划」章节，询问："阶段划分是否合理？时间估算是否现实？"

#### 章节 8：验收标准（可选）

**内容要点**：

- 功能验收（3-5 条）
- 性能验收（2-3 条）
- 兼容性验收（2-3 条）

**呈现方式**：如果使用场景需要验收标准，呈现此章节。

#### 章节 9：后续规划（可选）

**内容要点**：

- 短期优化（2-3 条）
- 长期规划（2-3 条）

**呈现方式**：如果使用场景需要后续规划，呈现此章节。

### 步骤 4：文档化（Documentation）

**在对话确认所有章节后**，将完整文档写入文件：

1. **保存位置**：
   - `docs/plans/YYYY-MM-DD-<feature-name>-prototype-design.md`

2. **文档结构**：

   ```markdown
   # [Feature Name] - 原型设计文档
   
   > **文档用途**：[使用场景]  
   > **创建日期**：YYYY-MM-DD  
   > **文档状态**：原型设计阶段
   
   ---
   
   ## 一、项目概述
   ...
   
   ## 二、系统架构
   ...
   
   ## 三、核心功能
   ...
   
   [其他章节...]
   
   ---
   
   **文档版本**：v1.0  
   **最后更新**：YYYY-MM-DD  
   **维护人**：[团队名称]
   ```

3. **内容要求**：
   - 精炼：每章 200-300 字，避免冗长
   - 清晰：使用列表、表格、图表增强可读性
   - 重点突出：核心价值、架构、关键功能
   - 避免技术细节：不包含具体代码、文件路径、API 接口定义

## 关键原则

参考 `brainstorming` skill 的原则：

- **一次一个问题** - 逐步细化，不要一次性问多个问题
- **分节呈现** - 每章 200-300 字，每章后确认
- **灵活迭代** - 如果某部分不对，回到对应步骤重新讨论
- **场景驱动** - 根据使用场景调整文档重点和详细程度
- **精炼优先** - 突出核心价值，避免技术细节

## 输出格式约定

### 文档结构（根据使用场景调整）

**必须包含的章节**：

1. 项目概述
2. 系统架构
3. 核心功能
4. 实施计划

**可选章节**（根据使用场景）：

- 数据结构设计（技术团队、详细评审）
- 关键业务规则（产品团队、业务评审）
- 用户界面设计（产品团队、UI 评审）
- 验收标准（技术团队、测试团队）
- 后续规划（长期规划、路线图）

### 内容精炼要求

- **项目概述**：必须使用四段式结构（定义、问题、解决、价值），每段 50-150 字，总计 400-600 字
  - 定义：50-100 字
  - 问题：100-150 字
  - 解决：100-150 字
  - 价值：100-150 字（3-5 条，用 ✅ 标记）
- **系统架构**：200-300 字，包含架构图或流程图
- **核心功能**：每个功能 50-100 字，列出 3-7 个
- **实施计划**：每个阶段 50-100 字，列出 3-5 个阶段

### 不包含内容

- 具体代码实现
- 详细的 API 接口定义
- 具体的文件路径
- 详细的测试用例
- 琐碎的技术细节

## 最佳实践与使用建议

- **何时使用本 Skill**：
  - 在原子任务文档完成后，需要生成汇报文档时
  - 在项目评审前，需要准备设计概览时
  - 在向利益相关者展示时，需要精炼的设计文档时

- **如何迭代**：
  - 采用对话式流程，逐步细化每个章节
  - 可以根据反馈调整文档重点和详细程度
  - 可以针对不同使用场景生成多个版本

- **如何与其它 Skill 协作**：
  - **推荐路径**：`pdd-skill1_7` → `pdd-skill1_8` → `pdd-skill1_10`（本 Skill，生成汇报文档）
  - **备选路径**：`pdd-skill1_7` → `pdd-skill1_8` → `pdd-skill1_9`（生成详细实现规划）
  - 两个路径可以并行，生成不同用途的文档

## 示例输出

```markdown
# 菜单版本化系统 - 原型设计文档

> **文档用途**：开会汇报用  
> **创建日期**：2024-12-19  
> **文档状态**：原型设计阶段

## 一、项目概述

### 1.1 定义（What）

菜单版本化系统是一个支持多租户 SaaS 系统中菜单和按钮权限多版本共存与切换的管理系统。核心概念包括：
- **版本映射表**：独立存储菜单和按钮权限的版本信息
- **租户版本管理**：每个租户使用一个全局菜单版本号
- **版本切换**：切换租户使用的版本时，自动更新所有菜单和按钮权限的显示

### 1.2 问题（Why）

**核心痛点**：
- 现有系统无法支持菜单的多版本共存，不同租户无法使用不同版本的菜单
- 无法灵活切换租户使用的菜单版本，升级或回退需要大量手动操作
- 菜单和按钮权限的版本信息耦合在业务表中，难以管理和维护

**典型场景**：
- 租户 A 需要使用菜单的 v1.0 版本（旧版界面）
- 租户 B 需要使用菜单的 v2.0 版本（新版界面）
- 同一租户在不同时期可能需要切换版本（如测试新版本后回退）

### 1.3 解决（How）

通过引入**版本映射表**，实现菜单和按钮权限的版本化管理：
- 每个菜单/按钮权限可以有多个版本配置（存储在版本映射表中）
- 每个租户使用一个全局版本号（存储在租户配置中）
- 切换版本时，系统自动查询版本映射表，更新菜单表的组件路径和接口URL，并刷新前端显示

**关键技术手段**：
- 版本映射表独立存储，与业务数据解耦
- 租户级版本控制，简化管理
- 向下兼容，现有菜单表结构不变

### 1.4 价值（Value）

- ✅ **多版本共存**：支持同一菜单的多个版本同时存在，满足不同租户需求
- ✅ **灵活切换**：租户可以快速切换使用的版本，无需手动修改配置
- ✅ **解耦设计**：版本信息独立存储，不影响现有菜单表结构，便于维护
- ✅ **向下兼容**：现有功能不受影响，支持平滑升级
- ✅ **可扩展性**：支持未来版本回滚、版本预览等高级功能
...
```

## 常见问题

**Q: 与 `pdd-skill1_9` 的区别是什么？**  
A: `pdd-skill1_9` 生成详细的实现规划（包含技术细节、文件路径、验收标准），适合开发团队。`pdd-skill1_10` 生成精炼的设计文档（突出价值、架构、功能），适合汇报和评审。

**Q: 可以基于原型代码生成文档吗？**  
A: 本 Skill 专注于基于原子任务文档生成文档。如果需要基于原型代码，请使用 `pdd-skill1_9-proto2planning`。

**Q: 如何调整文档的详细程度？**  
A: 通过指定不同的使用场景，Skill 会自动调整文档重点和详细程度。也可以通过在对话中明确要求来调整。
