# 将想法通过头脑风暴转化为设计

## 概览

通过自然的协作对话，将想法转化为完全成型的设计和规范。

首先了解当前的项目背景，然后逐个提出问题以细化想法。一旦你理解了要构建的内容，就分小节（200-300字）展示设计，并在每一节后检查是否符合预期。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近的提交）
- 逐个提问以细化想法
- 尽可能优先使用多项选择题，但也接受开放式问题
- 每条消息只问一个问题——如果一个话题需要更多探索，将其拆分为多个问题
- 专注于理解：目的、约束、成功标准

**探索方法：**
- 提出 2-3 种不同的方法并分析权衡
- 以对话方式展示选项，并附上你的推荐和理由
- 首先提出你推荐的选项并解释原因

**展示设计：**
- 一旦你认为你理解了要构建的内容，就开始展示设计
- 将其拆分为 200-300 字的章节
- 在每一节后询问目前为止看起来是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 随时准备回头澄清如果不合理的地方

## 设计之后

**文档化：**
- 将经过验证的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely 技能
- 将设计文档提交到 git

**实施（如果继续）：**
- 询问：“准备好开始实施设置了吗？”
- 使用 superpowers:using-git-worktrees 创建隔离的工作区
- 使用 superpowers:writing-plans 创建详细的实施计划

## 关键原则

- **一次一个问题** - 不要用多个问题轰炸用户
- **首选多选题** - 可能的话，比开放式问题更容易回答
- **无情地执行 YAGNI** - 从所有设计中移除不必要的功能
- **探索替代方案** - 在定稿前总是提出 2-3 种方法
- **增量验证** - 分节展示设计，逐节验证
- **保持灵活** - 当事情不合理时，回头并澄清
