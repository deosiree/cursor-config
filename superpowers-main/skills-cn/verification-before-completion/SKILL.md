# 完成前验证

## 概述

在没有验证的情况下声称工作完成是不诚实，不是效率。

**核心原则：** 始终先有证据，再有声明。

**违反此规则的文字就是违反此规则的精神。**

## 铁律

```
没有新的验证证据，不进行完成声明
```

如果你在此消息中没有运行验证命令，你不能声称它通过。

## 门控函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令证明此声明？
2. 运行：执行完整命令（新的、完整的）
3. 阅读：完整输出，检查退出代码，计算失败
4. 验证：输出是否确认声明？
   - 如果否：用证据说明实际状态
   - 如果是：用证据说明声明
5. 只有这样：进行声明

跳过任何步骤 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不足 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 之前的运行，"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查，推断 |
| 构建成功 | 构建命令：退出 0 | Linter 通过，日志看起来好 |
| 错误修复 | 测试原始症状：通过 | 代码更改，假设修复 |
| 回归测试有效 | 红绿循环验证 | 测试通过一次 |
| 代理完成 | VCS diff 显示更改 | 代理报告"成功" |
| 满足需求 | 逐行清单 | 测试通过 |

## 危险信号 - 停止

- 使用"应该"、"可能"、"似乎"
- 在验证之前表达满意（"太好了！"、"完美！"、"完成！"等）
- 即将提交/推送/PR 而不验证
- 信任代理成功报告
- 依赖部分验证
- 认为"就这一次"
- 累了想要工作结束
- **任何暗示成功而没有运行验证的措辞**

## 合理化预防

| 借口 | 现实 |
|--------|---------|
| "现在应该有效" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 无例外 |
| "Linter 通过" | Linter ≠ 编译器 |
| "代理说成功" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查足够" | 部分证明不了什么 |
| "不同措辞所以规则不适用" | 精神高于文字 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过" / "看起来正确"
```

**回归测试（TDD 红绿）：**
```
✅ 编写 → 运行（通过）→ 恢复修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我编写了回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出 0] "构建通过"
❌ "Linter 通过"（linter 不检查编译）
```

**需求：**
```
✅ 重新阅读计划 → 创建清单 → 验证每个 → 报告空白或完成
❌ "测试通过，阶段完成"
```

**代理委托：**
```
✅ 代理报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任代理报告
```

## 为什么这很重要

来自 24 个失败记忆：
- 你的合作伙伴说"我不相信你" - 信任被打破
- 未定义的函数被发布 - 会崩溃
- 缺少需求被发布 - 不完整的功能
- 时间浪费在虚假完成 → 重定向 → 返工
- 违反："诚实是核心价值。如果你撒谎，你将被替换。"

## 何时应用

**总是在之前：**
- 任何成功/完成声明的变体
- 任何满意表达
- 关于工作状态的任何积极声明
- 提交、PR 创建、任务完成
- 移动到下一个任务
- 委托给代理

**规则适用于：**
- 确切短语
- 释义和同义词
- 成功的含义
- 任何暗示完成/正确性的通信

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

这是不可协商的。
